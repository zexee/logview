# lv 新开发计划

## 开发原则

- 删除现有实现思路，按新的模块边界重新开发。
- 每个阶段必须有可运行的程序、单元测试或性能测试作为验收依据。
- 先做核心数据模型和过滤引擎，再做 UI，再做组合逻辑。
- 后台过滤和前台显示分离，任何大文件计算不能阻塞 UI 主循环。
- 文件读取、行索引、过滤 bitmap、显示 buffer 都要面向大文件设计，避免按行复制完整内容。

## 阶段 1：规则过滤核心模块

### 目标

实现一个独立模块，读取规则集和输入文件，生成：

- 每条规则对应的中间 bit array
- 最终显示用 bit array

bit 含义：

- `1` 表示该行显示或通过当前规则层
- `0` 表示该行隐藏或不再进入下一层

### 过滤语义

- 规则按顺序组成 pipeline。
- 第 1 层处理所有行。
- 第 N 层只处理第 N-1 层 bit 为 `1` 的行。
- 过滤不能先做完整一层再做下一层。
- 必须按行处理：读取一行后，立即依次执行所有规则，并设置所有规则 bitmap 的对应 bit，最后设置最终 bitmap。

示例：

```text
line 0 -> rule 0 -> rule 1 -> rule 2 -> set final bit
line 1 -> rule 0 -> rule 1 -> rule 2 -> set final bit
```

### 模块设计

- `MappedFile`：使用 mmap 或普通文件映射接口读取文件。
- `LineIndex`：预计算每行起止位置，支持 O(1) 或接近 O(1) 获取行内容。
- `BitArray`：紧凑位图，底层使用 `std::vector<uint64_t>`。
- `Rule`：单条规则，支持显示/隐藏语义。
- `RuleSet`：规则集合，支持读取、保存、增删、移动、编辑。
- `FilterEngine`：按行执行规则 pipeline，输出 `FilterResult`。
- `FilterResult`：保存所有中间 bitmap 和最终 bitmap。

### 第一版规则格式

先使用简单文本格式，便于调试：

```text
ss ERROR
hh DEBUG
s user=[0-9]+
```

字段：

- 第 1 列：`s`、`h`、`ss` 或 `hh`
- 第 2 列到行尾：匹配表达式

操作符：

- `s`：show regex
- `h`：hide regex
- `ss`：show literal
- `hh`：hide literal

后续可以扩展 `prefix`、`suffix`、大小写选项等。

### 单元测试

必须覆盖：

- 空规则集：所有行可见。
- 单个 `ss`。
- 单个 `hh`。
- 多层规则：下一层只处理上一层 bit 为 `1` 的行。
- 中间 bit array 内容正确。
- 最终 bit array 内容正确。
- 规则文件读取和保存。
- 空文件、无结尾换行文件、超长行文件。
- regex 规则合法和非法输入。

### 性能测试

提供独立可执行程序：

```bash
./build/perf_filter_engine <log_file> <rule_file>
```

输出：

- 文件大小
- 行数
- 规则数量
- 总耗时
- 每秒处理行数
- 每秒处理字节数
- bitmap 内存占用

优化方向：

- 行内容使用 `std::string_view`，不复制行。
- literal 匹配优先使用 `std::string_view::find`。
- bitmap 按 `uint64_t` 写入。
- 规则对象预编译 regex。
- 避免虚函数热路径；第一版可以用简单枚举分发。

### 阶段验收

- `test_filter_engine` 全部通过。
- `perf_filter_engine` 可运行并输出稳定指标。
- 核心模块不依赖 ncurses。

## 阶段 2：UI 基础框架

### 目标

实现 ncurses TUI 基础能力：

- 窗口管理
- 上下分割窗口
- 控件焦点
- 键盘输入分发
- 底部行编辑区域

### 窗口布局

第一版布局：

```text
+-------------------------------+
| main log window                |
|                               |
+-------------------------------+
| rule window                    |
+-------------------------------+
| : command / line editor        |
+-------------------------------+
```

### 控件

- `Screen`：ncurses 生命周期和终端尺寸。
- `Window`：窗口矩形、边框、刷新。
- `SplitLayout`：上下分割布局。
- `FocusManager`：当前焦点窗口。
- `LineEditor`：底部命令行编辑器。

### LineEditor 行编辑功能

支持：

- 普通字符输入。
- 左右移动光标。
- Backspace/Delete。
- Home/End。
- Enter 提交。
- Esc 取消。

调用方式：

- 命令模式：用户按 `:`，底部进入 `:xxx` 输入。
- 规则编辑：用户在规则窗口按 Enter，把当前规则载入底部编辑区，编辑完成后 Enter 替换回规则窗口。

### 阶段验收

- 程序启动后能显示上下分割窗口和底部命令行。
- 能用键盘在窗口间切换焦点。
- `:` 能进入命令输入。
- 规则窗口按 Enter 能进入规则编辑输入。
- 不要求真正过滤和渲染日志内容。

## 阶段 3：窗口渲染和显示 buffer

### 目标

实现窗口内容渲染，主要能力是按行显示文件内容。

### 显示能力

- 支持长行换行显示。
- 光标所在行高亮。
- 支持上下滚动。
- 支持显示行号。
- 支持高亮某些字符范围。
- 窗口只显示内容，不支持直接编辑。

### 后端 buffer 类型

显示窗口支持两类后端：

1. 普通文件 + 规则集显示  
   用于规则窗口或文本窗口。

2. 文件 + bit array 过滤显示  
   用于 log 显示窗口，只显示最终 bitmap 为 `1` 的行。

### 文件显示索引

文件载入后预计算：

- 每行起止位置。
- 长行在当前窗口宽度下的换行位置或可快速计算换行。
- 可见逻辑行到物理文件行的映射。

### 滚动语义

- 上下方向按显示行移动。
- 光标移动时窗口跟随滚动。
- 过滤模式下跳过 bitmap 为 `0` 的文件行。

### 阶段验收

- 能打开一个 log 文件并显示。
- 长行能换行。
- 上下移动光标时高亮正确。
- 能显示行号。
- 给定 bit array 后只显示 bit 为 `1` 的行。

## 阶段 4：组合过滤和显示

### 目标

把阶段 1 的过滤引擎和阶段 3 的显示窗口组合起来：

- 启动时载入预设规则文件和 log 文件。
- 后台执行过滤。
- 前台显示日志。
- 支持上下滚动。

### 后台过滤

- UI 主线程只负责输入和渲染。
- 过滤线程生成新的 `FilterResult`。
- 过滤完成后以快照方式替换显示窗口的 bitmap。
- 过滤期间继续显示旧结果或未过滤内容。

### 阶段验收

- 启动命令：

```bash
./build/lv <log_file> <rule_file>
```

- 程序能显示 log。
- 后台过滤完成后，显示结果刷新为最终 bitmap。
- 过滤期间 UI 可滚动和响应键盘。

## 阶段 5：规则窗口

### 目标

实现规则列表窗口，支持查看和修改规则。

### 功能

- 显示规则列表。
- 当前规则高亮。
- 删除规则。
- 上移/下移规则。
- 编辑规则。
- 新增规则。
- 每次规则改动后触发后台重新过滤。
- 过滤完成后刷新 log 显示。

### 编辑流程

- 规则窗口按 Enter。
- 当前规则文本载入底部 `LineEditor`。
- 用户编辑后按 Enter。
- 解析规则，成功则替换原规则并触发过滤。
- 解析失败则保留原规则并显示错误状态。

### 阶段验收

- 能在 UI 内删除、移动、编辑规则。
- 修改规则后 log 显示会重新过滤。
- UI 不因重新过滤而阻塞。

## 阶段 6：命令模式和文件操作

### 目标

支持 vi 风格命令模式，用于规则集和过滤结果文件操作。

### 命令

第一版支持：

```text
:open <log_file>
:rules <rule_file>
:write-rules [rule_file]
:save-filtered <output_file>
:quit
```

### 行为

- `:open` 载入新的 log 文件，并重新过滤。
- `:rules` 载入规则集，并重新过滤。
- `:write-rules` 保存当前规则集。
- `:save-filtered` 将当前最终 bitmap 为 `1` 的日志行另存为文件。
- `:quit` 退出程序。

### 阶段验收

- 命令模式可操作规则集文件。
- 可保存过滤后的日志文件。
- 错误命令或文件错误能在状态栏显示。

## 最终验收

- 大文件载入后 UI 不阻塞。
- 规则过滤按行 pipeline 处理，而不是按层批处理。
- 每条规则有独立中间 bit array。
- 最终 bit array 正确。
- log 窗口支持长行换行、行号、光标高亮、滚动。
- 规则窗口支持删除、移动、编辑。
- 命令模式支持载入/保存规则集和另存过滤结果。
- 单元测试和性能测试可在本地一键运行。

## 推荐执行顺序

1. 清理旧代码结构，只保留文档和构建入口。
2. 实现阶段 1 核心过滤模块。
3. 写完阶段 1 单元测试和性能测试。
4. 阶段 1 测试通过后再进入 UI。
5. UI 每完成一层能力都保持可运行状态。
